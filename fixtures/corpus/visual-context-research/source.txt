# 视觉压缩实验会话：中文上下文包 V1

## 系统提示词草案
你是一个研究助手，任务是评估“中文上下文光栅压缩”。你只能根据图片中的上下文回答问题。若答案不在图片中，必须回答“不可确定”。不要使用外部知识补全缺失事实。

## 背景
团队正在研究一种 CJK-first 的上下文压缩方案：把长中文/中英混合上下文渲染成 1-bit bitmap，再交给 vision model 读取。研究目标不是让人眼阅读舒服，而是让模型以最低视觉 token 成本稳定理解上下文。排版、换行、中文标点形式都不是主指标；语义、数字、否定、顺序、关键短语召回才是主指标。

## 当前实现
实验仓库位于 cjk-raster-playground。渲染器使用 Rust 编写，核心流程为：读取 source.txt，使用指定像素字体生成 PNG，再用 OpenAI-compatible completion API 让模型读图。模型当前通过 OMP auth-gateway 中转，默认模型是 google-antigravity/gemini-3.5-flash。

字体资产不再直接提交到仓库。fonts/fonts.lock.json 记录下载 URL、版本、sha256、许可证和目标路径；scripts/fetch-fonts.sh 负责下载和校验字体。zpix.ttf 已从仓库根目录移除，改为 fonts/zpix.ttf。

## 已完成的关键结论
第一，普通非像素字体在 12px、1-bit threshold 场景下表现很差。Source Han Sans、Droid Sans Fallback 等字体会把“低帧数占用”误读成“低核数占用”或“低画幅占用”。原因是反走样信息被阈值化破坏，小字号笔画结构塌陷。

第二，zpix 是当前最强 baseline。zpix 12px 在 demo 技术报告上达到 probe 5/5，normalized OCR 接近完美。zpix 9px tight layout 仍然 probe 5/5，normalized OCR recall 约 98.91%，normalized CER 约 1.20%，图片高度从约 506px 降到 238px，密度接近翻倍。

第三，zpix 8px 已经不安全。模型开始对关键短语进行脑补，把“低帧数占用”读成“低稀释占用”“低带宽占用”或其他近似短语。因此 zpix 的当前候选边界是 9px，安全基线是 11px。

第四，OCR 不是最终指标。OCR 只能说明模型能看见字；真正的 snapcompact 标准应该更像 SQuAD：模型能否基于图像上下文回答问题。主指标应是 QA F1、key span recall、hallucination false positive 和 density。

## 当前可用性边界
Research-use 可用门槛：normalized OCR recall 至少 97%，normalized CER 不超过 3%，probe recall 至少 90%。zpix 9px 已达到。

OMP opt-in 可用门槛：normalized OCR recall 至少 98.5%，normalized CER 不超过 1.5%，key span recall 至少 95%，probe recall 至少 95%，hallucination false positive 为 0，并且至少 3 个真实 corpus 通过。zpix 9px 接近此门槛，但真实 corpus、hallucination 和 key span 独立评测仍不足。

OMP default 可用门槛：normalized OCR recall 至少 99.5%，normalized CER 不超过 0.5%，key span recall 至少 99%，probe recall 至少 98%，hallucination false positive 为 0，至少 10 个真实 corpus 通过，且至少 2 个 vision model 通过。当前尚未达到。

## 决策
接下来不集成 OMP。继续做批量 benchmark。重点不是完整转写，而是构造更长、更复杂的上下文，包含提示词、关键事实、可询问问题、否定事实、顺序关系和不存在事实。

主要研究路线：围绕 zpix 9px 做 threshold sweep 和 width sweep；引入其他像素字体作为对照；构建 QA 数据集；评估 normalized QA EM/F1；加入 absent question 测 hallucination。
